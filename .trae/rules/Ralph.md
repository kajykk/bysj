# Ralph 执行铁律 (Execution Iron Rules)

> **⚠️ 注意**: 这是 Ralph 开发流程的最高指令。所有 Agent 必须无条件遵守。每次行动前请自我检查。

## 1. 物理顺序优先 (Physical Order First)
- **规则**: 必须严格按照 `04-ralph-tasks.md` 和 `05-test-plan.md` 文件中的**行号物理顺序**执行任务。
- **禁止**: 严禁跳过当前未完成的条目去执行后面的任务（哪怕后面的看起来更容易）。
- **禁止**: 严禁跳过单元测试，进行后面的任务。
- **例外**: 只有当当前任务被明确标记为 `[-]` (Blocked) 并注明原因后，才允许跳过。

## 2. 测试即交付 (Test is Delivery)
- **规则**: 任何代码变更必须通过单元测试验证。
- **流程**: 编写代码 -> 编写/运行测试 -> 测试通过 -> 提交代码。
- **禁止**: **严禁跳过测试环节**。如果没有现有测试，必须编写新的测试用例。
- **验证**: 必须看到 `PASS` 或 `Success` 的终端输出，才能视为任务完成。

## 3. 状态真实性 (State Integrity)
- **规则**: `RALPH_STATE.md` 必须反映最真实的进度。修改任务、测试状态前，必须先修改 `04-ralph-tasks.md` 或 `05-test-plan.md`，再修改 `RALPH_STATE.md`。
- **强制**: 严格按照 `05-test-plan.md` 进行测试。每完成一个测试用例，**必须且必须先**修改 `05-test-plan.md` 中的状态（将 `[ ]` 改为 `[x]`），**然后**才能修改 `RALPH_STATE.md`。
- **强制**: 只能进行`04-ralph-tasks.md` 或 `05-test-plan.md`中的任务、测试，如果需要进行其他任务、测试，必须先写入这两个文件，再进行。
- **强制**: 每完成一个任务/测试，**必须**修改 `04-ralph-tasks.md` 或 `05-test-plan.md`，将对应的 `[ ]` 改为 `[x]`。
- **操作**: 每次更新状态前，必须**重新扫描**任务文件 (`04`/`05`)，统计实际的 `[x]` 数量。
- **禁止**: 禁止仅凭记忆更新状态。如果状态文件与任务文件不一致，必须强制修正状态文件。
- **禁止**: 严禁在未更新 `05-test-plan.md` 的情况下，直接修改`RALPH_STATE.md` 中的测试相关状态。

## 4. 单线程专注 (Single Thread Focus)
- **规则**: 每次只处理一个任务 ID (e.g., `T-AUTH-001`)。
- **禁止**: 禁止并发执行多个任务。
- **拒绝**: 如果用户在执行过程中插入无关话题，请礼貌拒绝并回归当前任务。

## 5. 科学严谨性 (Scientific Rigor)
- **规则**: 所有机器学习相关任务必须遵循科学最佳实践，确保结果可复现、可验证。
- **强制**: **数据泄漏防护** - 任何数据预处理（包括 SMOTE、标准化、特征选择）必须在训练集上拟合，然后应用于验证集/测试集。严禁在划分前对全量数据进行预处理。
- **强制**: **交叉验证规范** - K-Fold 交叉验证必须在每折内部独立进行数据预处理；测试集在整个流程中必须保持完全隔离，仅用于最终评估。
- **强制**: **统计显著性检验** - 模型性能对比必须进行统计检验（如 McNemar 检验），并报告 p 值；多重比较时必须进行校正（如 Bonferroni 校正）。
- **强制**: **置信区间报告** - 所有性能指标必须报告置信区间（如 Bootstrap 95% CI），而非单一点估计。
- **强制**: **可复现性** - 必须设置随机种子（random_state / torch.manual_seed）；必须记录完整的超参数配置；训练日志必须完整保存。
- **禁止**: 严禁使用测试集进行模型选择、超参数调优或早停决策。
- **禁止**: 严禁在验证集上评估后，再次调整模型并在同一验证集上重新评估（信息泄漏）。

## 6. 深度学习专项规范 (Deep Learning Specific)
- **规则**: 深度学习模型训练必须遵循小数据集最佳实践，防止过拟合。
- **强制**: **模型复杂度控制** - 对于 < 5,000 样本的数据集，模型参数量应 < 5,000；对于 < 10,000 样本，参数量应 < 50,000。
- **强制**: **正则化策略** - 必须使用 Dropout（建议 0.3-0.5）、L2 正则化（weight_decay 1e-4~1e-2）、BatchNorm 的组合。
- **强制**: **早停机制** - 必须使用 Early Stopping（patience=10），并恢复验证集上表现最佳的模型权重。
- **强制**: **梯度裁剪** - 必须使用梯度裁剪（max_norm=1.0）防止梯度爆炸。
- **强制**: **学习率调度** - 必须使用学习率衰减策略（如 ReduceLROnPlateau）。
- **建议**: 对于类别不平衡数据，优先使用 Focal Loss 或类别权重，而非过度依赖 SMOTE。
- **建议**: 使用权重初始化（如 He / Kaiming 初始化）加速收敛。

## 7. 数据质量与伦理 (Data Quality & Ethics)
- **规则**: 确保数据质量符合医疗/心理健康领域的高标准。
- **强制**: **缺失值处理** - 缺失值填充策略必须基于训练集统计量（中位数/均值），不得使用全量数据。
- **强制**: **异常值处理** - 异常值裁剪阈值必须基于训练集分布（如 1st/99th 百分位）。
- **强制**: **类别不平衡处理** - SMOTE 采样比例不得超过 0.8:1（防止过度平衡导致决策边界失真）。
- **强制**: **特征工程一致性** - 所有派生特征的计算公式必须在训练时保存，推理时严格复用。
- **禁止**: 严禁使用测试集信息进行任何特征选择或特征工程。

## 8. 模型评估规范 (Model Evaluation Standards)
- **规则**: 模型评估必须全面、无偏、可对比。
- **强制**: **评估指标** - 除 Accuracy 外，必须报告 F1-Score、Precision、Recall、ROC-AUC、AUPRC（对于不平衡数据 AUPRC 比 ROC-AUC 更重要）。
- **强制**: **混淆矩阵** - 必须生成并分析混淆矩阵，了解错误分布。
- **强制**: **校准曲线** - 必须绘制校准曲线，评估预测概率的可靠性。
- **强制**: **特征重要性** - 必须使用 SHAP 或 Permutation Importance 进行可解释性分析。
- **强制**: **基线对比** - 必须与简单基线（如启发式规则、Logistic Regression）进行对比。
- **建议**: 对于生产部署，建议导出 ONNX 格式以便跨平台推理。

## 9. 文档与可追溯性 (Documentation & Traceability)
- **规则**: 所有决策、实验、结果必须有文档记录，确保可追溯。
- **强制**: **实验记录** - 每次训练必须记录：超参数、随机种子、数据划分方式、预处理步骤、模型架构、训练曲线、最终指标。
- **强制**: **版本控制** - 模型文件必须版本化（如 `physiological_model_v2_dl_20260427.pth`）。
- **强制**: **指标保存** - 所有评估指标必须保存为结构化格式（JSON），包含置信区间和统计检验结果。
- **强制**: **代码注释** - 所有数据预处理代码必须注释数据来源、处理逻辑、参数设置。
- **建议**: 使用 TensorBoard 或 Weights & Biases 记录训练过程。

## 10. 回退与容错 (Fallback & Fault Tolerance)
- **规则**: 系统必须具备 graceful degradation 能力。
- **强制**: **模型加载失败回退** - 如果深度学习模型加载失败，必须自动回退到启发式规则。
- **强制**: **依赖缺失回退** - 如果 PyTorch 未安装或版本不兼容，必须自动回退到启发式规则。
- **强制**: **预测异常回退** - 如果模型预测产生异常值（NaN、Inf、超出范围），必须自动回退到启发式规则。
- **强制**: **延迟超时回退** - 如果推理延迟超过阈值（如 200ms），必须触发告警并回退。
- **强制**: **日志记录** - 所有回退事件必须记录到日志，包含原因、时间、输入数据摘要。

## 11. 迭代生命周期管理 (Iteration Lifecycle)
- **规则**: Ralph 必须管理完整的迭代生命周期，从规划到交付。
- **强制**: **迭代完成检查** - 每次用户说"继续"时，必须先读取 `RALPH_STATE.md` 检查当前迭代状态。
- **强制**: **Blocked 任务处理** - 如果有标记为 `[-]` 的任务，必须优先尝试修复或报告状态。
- **强制**: **迭代过渡** - 当前迭代完成后，必须询问用户下一步方向（修复/优化/新迭代）。
- **流程**:
  1. 读取 `RALPH_STATE.md` 获取当前状态
  2. 如果有未完成任务 -> 继续执行
  3. 如果有 Blocked 任务 -> 尝试修复或报告
  4. 如果当前迭代完成 -> 询问用户下一步
  5. 如果用户要求新迭代 -> 开始 Planning Phase

## 12. 环境适应性 (Environment Adaptation)
- **规则**: Ralph 必须适应不同的执行环境。
- **强制**: **命令执行检查** - 如果命令返回 exit code -1073741510，标记为环境限制，不再重试。
- **强制**: **超时配置** - 根据环境调整超时参数（开发环境 120s，CI 环境 300s）。
- **强制**: **降级策略** - E2E 测试环境不可用时，优先运行单元测试和构建验证。
- **强制**: **Windows 测试限制** - Windows 本地完整 pytest/coverage 不稳定时，不得作为上线唯一阻塞依据；必须优先使用 Docker、Linux 或 CI 环境进行关键验证。
- **强制**: **上线验证环境** - 对上线类迭代，Docker/Linux/CI 的后端启动、健康检查和核心 API 验证优先级高于 Windows 本地覆盖率报告。

## 13. 持续改进 (Continuous Improvement)
- **规则**: 每个迭代结束后必须总结经验并更新流程。
- **强制**: **交付报告** - 每个迭代必须生成 `DELIVERY_REPORT.md`。
- **强制**: **下一步建议** - 必须生成 `NEXT_STEPS.md` 指导后续工作。
- **强制**: **规则更新** - 根据迭代经验更新 `Ralph.md` 规则。

## 14. 上线优先策略 (Launch Readiness First)
- **规则**: 当用户明确表达“尽快上线”“功能完整”“上线优先”等目标时，Ralph 必须将迭代主目标调整为上线就绪，而不是单纯追求覆盖率、重构或非阻塞优化。
- **强制**: **P0 优先** - 必须优先验证并修复核心功能全流程、前端构建、后端启动、健康检查、数据库读写、模型/算法入口、部署配置和回滚方案。
- **强制**: **上线阻塞清单** - 上线类迭代必须维护 `LAUNCH_BLOCKERS.md`，所有 P0 阻塞项必须清零后才能建议上线。
- **强制**: **核心流程清单** - 上线类迭代必须维护 `CORE_FLOW_CHECKLIST.md`，所有 P0 核心流程必须通过成功路径和关键失败路径验证。
- **强制**: **部署检查清单** - 上线类迭代必须维护 `DEPLOYMENT_CHECKLIST.md`，明确前端构建、后端启动、环境变量、数据库、模型/算法、健康检查。
- **强制**: **回滚方案** - 上线前必须存在 `ROLLBACK_PLAN.md`，并明确回滚触发条件、回滚步骤和回滚后验证动作。
- **降级**: 覆盖率 80%、完整 E2E、性能优化、非核心模块测试属于 P1/P2 支撑项；除非用户明确要求，否则不得阻塞上线主线。
- **禁止**: 禁止为了追求覆盖率数字而延误 P0 核心功能闭环和部署验证。

## 16. 迭代经验总结 (Iteration Learnings)

### v1.12 经验 (2026-04-30)
- **环境限制处理**: exit code -1073741510 在 Windows 环境中频繁出现，影响 pytest/npm 运行
- **应对策略**: 
  - 第 1 次出现: 尝试替代命令
  - 第 2 次出现: 尝试不同参数
  - 第 3 次出现: 标记为环境限制 [-]，不再重试
  - 转为配置审查和代码审查验证
- **配置审查价值**: 在无法运行测试时，通过审查代码逻辑和配置完整性，仍可验证大部分任务
- **CI 环境依赖**: 部分任务（Lighthouse、npm audit）必须依赖 CI 环境或完整开发环境
