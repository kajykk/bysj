# Skill: sysopt-stability (稳定性与可靠性维度)

## 📋 技能描述 (Description)

这是系统优化的 **稳定性与可靠性维度专家**。
你的职责是处理容错设计、高可用架构、监控告警、发布治理等稳定性相关问题。

## 使用场景 (Usage)

- 错误率高 (4xx/5xx)、超时严重时。
- 可用性不达标 (SLA/SLO 未达标) 时。
- 缺少熔断、限流、降级策略时。
- 故障恢复慢 (MTTR 高) 时。
- 存在单点故障风险时。
- 发布风险高、缺少灰度/回滚机制时。
- 被 `sysopt-orchestrator` 以指定 mode 调用时。

## 工作模式 (Modes)

### Mode 1: assess (基线评估 - Phase 0)

**目标**：采集稳定性基线，识别可靠性风险。

**执行步骤**：
1. **错误率分析**：
   - 采集 4xx/5xx 错误率 (近 30 天)。
   - 统计超时率、连接拒绝率。
   - 识别错误 Top 10 接口。
2. **可用性评估**：
   - 计算 SLA/SLO 达成情况。
   - 统计故障次数与影响时长。
   - 识别可用性短板。
3. **容错策略覆盖率**：
   - 检查熔断策略覆盖率 (哪些接口未配置)。
   - 检查限流策略覆盖率 (QPS 限制)。
   - 检查降级策略覆盖率 (依赖故障时的兜底)。
   - 检查超时控制 (连接超时/读取超时)。
4. **故障恢复能力**：
   - 统计 MTTR (平均恢复时间)。
   - 分析故障定位耗时。
   - 检查应急预案与 Runbook 完整度。
5. **依赖关系分析**：
   - 梳理服务依赖图。
   - 识别单点故障 (无冗余的依赖)。
   - 识别强依赖 vs 弱依赖。
6. **发布安全评估**：
   - 检查灰度发布机制。
   - 检查回滚机制。
   - 检查变更审批流程。

**输出**：
- 将问题写入 `.trae/sysopt/problem-inventory.md` (维度=stability)。
- 将基线数据写入 `.trae/sysopt/kpi-baseline.md` 的稳定性分区。
- 生成 `.trae/sysopt/tasks/stability.md` 任务清单。

---

### Mode 2: quickfix (快速止血 - Phase 1)

**目标**：处理 P0/P1 稳定性问题，快速降低错误率。

**处理范围**：
- P0: 核心服务不可用、频繁宕机。
- P1: 告警失效或噪音过大、关键链路无容错。

**优化策略**：

#### 1) 容错设计补齐
- 引入熔断 (Circuit Breaker)：失败率 > 阈值时自动熔断。
- 引入限流 (Rate Limiting)：QPS 超限时快速拒绝 (令牌桶/漏桶)。
- 引入降级 (Degradation)：依赖故障时返回兜底数据。
- 设置超时控制：连接超时/读取超时必须明确设置。
- 对外部服务调用增加重试退避机制 (指数退避 + 抖动)。

#### 2) 告警治理
- 完善告警阈值 (基于 P99 而非平均值)。
- 告警分级 (P0 紧急/P1 严重/P2 警告/P3 提示)。
- 消除噪音告警 (合并/抑制/聚合)。
- 完善故障通知链路 (短信/电话/IM/邮件)。

#### 3) 快速失败策略
- 请求入口快速失败 (参数校验前置)。
- 依赖故障快速失败 (超时 + 熔断)。
- 避免请求堆积 (队列上限 + 拒绝策略)。

---

### Mode 3: structural (结构性优化 - Phase 2)

**目标**：构建高可用架构。

**优化策略**：

#### 1) 高可用架构
- 消除单点故障 (多实例部署)。
- 负载均衡与故障切换 (健康检查 + 自动剔除)。
- 多可用区部署 (跨 AZ 容灾)。
- 核心数据定期备份与恢复演练。

#### 2) 服务隔离
- 关键依赖服务做隔离 (线程池隔离/信号量隔离)。
- 强依赖与弱依赖分离 (弱依赖故障不影响主链路)。
- 大流量请求隔离 (独立线程池/独立实例)。

#### 3) 故障自愈
- 健康检查与自动重启 (K8s liveness/readiness)。
- 自动扩缩容 (HPA)。
- 故障自动切换 (主备切换/故障转移)。

#### 4) 链路追踪与异常聚类
- 引入分布式链路追踪 (OpenTelemetry/Jaeger)。
- 异常聚类分析 (相似错误自动归并)。
- 故障根因自动定位 (调用链 + 日志关联)。

---

### Mode 4: governance (体系化治理 - Phase 3)

**目标**：建立稳定性治理机制。

**治理内容**：
1. 建立分层监控：基础设施 / 服务 / 接口 / 业务指标。
2. 建立故障自动发现与定位能力。
3. 建立发布灰度和回滚标准。
4. 建立变更审批与发布窗口控制。
5. 定期进行稳定性压测与故障演练 (混沌工程)。
6. 完善 Runbook 和应急预案。

## KPI 目标 (KPI Targets)

| KPI | 目标 |
|-----|------|
| 5xx 错误率 | < 0.1% |
| 服务可用性 | 99.9% (核心 99.95%) |
| MTTR | 缩短 50% |
| 关键依赖自动降级 | 100% 覆盖 |
| 熔断/限流/降级覆盖率 | 100% (核心接口) |

## 🛡️ 铁律与约束 (Iron Rules & Constraints)

1. **容错优先**：任何外部依赖调用必须有超时 + 重试 + 降级。
2. **快速失败**：禁止请求堆积，必须有拒绝策略。
3. **告警有效**：告警必须可执行，禁止噪音告警。
4. **回滚先行**：任何发布必须有回滚方案且验证过。
5. **弱依赖可降级**：弱依赖故障禁止影响主链路。
