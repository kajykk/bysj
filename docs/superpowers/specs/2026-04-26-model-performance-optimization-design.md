# 模型性能优化设计稿

## 背景

当前模型性能评估报告显示：

- 结构化模型 F1 为 `0.8393`，低于基准 `0.85`
- 文本模型 F1 为 `0.9741`，已达标
- 多模态融合引擎平均延迟为 `156.07 ms`，但 `p95` 达到 `313.10 ms`，高于基准 `200 ms`
- 生理模型与融合引擎仅在 4 条样本上评估，结果参考价值有限，需要在后续评估中继续保留，但不能将其作为主要结论依据

本次优化目标不是全面重构系统，而是在尽量保持现有架构稳定的前提下，优先让关键指标达标：

1. 先把结构化模型 F1 提升到基准线以上
2. 同时降低融合引擎的端到端延迟，尤其压低尾延迟
3. 保留文本模型的现有优势，避免为了优化其它模块而引入不必要的回退

## 优化目标

### 必达目标

- 结构化模型测试集 F1 `>= 0.85`
- 融合引擎 `p95` 延迟 `<= 200 ms`
- 现有文本模型性能不下降，至少保持当前 F1 水平附近

### 次级目标

- 降低结构化模型的误报或漏报偏差，使精确率与召回率更平衡
- 降低融合路径中的极端慢请求，减少 `max_ms` 与 `p99` 的抖动
- 保持训练和推理逻辑可解释、可维护，避免引入过重复杂度

## 设计原则

1. **先修最便宜的提升点**：优先做阈值优化、特征清理、轻量参数搜索，这些通常能以最小代价换取最接近基准的提升。
2. **避免在请求热路径中做重计算**：融合延迟问题优先从工程路径优化，而不是直接换更复杂的在线模型。
3. **以验证集驱动决策**：所有阈值、特征和模型选择都应以验证集或交叉验证结果为依据，而不是凭经验拍板。
4. **保留回退能力**：任何新方案都应能回退到当前已知可运行的实现，确保交付稳定性。

## 方案总览

### 1. 结构化模型优化

结构化模型当前的主要问题是 F1 略低于基准。由于其召回率高于精确率，说明当前模型偏向“多报阳性”，因此优先优化方向应是：

- 特征层面：清理无效类别、增强派生特征、统一数值化处理
- 模型层面：扩展可选模型和参数搜索空间
- 阈值层面：在验证集上搜索最优分类阈值，而不是固定使用默认 `0.5`

#### 结构化训练流程调整

1. 保留现有数据清洗逻辑
2. 继续构建数值特征、类别特征和派生特征
3. 在训练过程中增加阈值搜索：
   - 对验证集概率输出扫描多个阈值
   - 以 F1 作为主目标，必要时辅以 precision/recall 平衡约束
4. 扩展模型搜索：
   - 保留 `LogisticRegression`
   - 保留 `RandomForestClassifier`
   - 保留 `CatBoostClassifier`
   - 必要时增加更适合表格任务的轻量提升模型候选

#### 结构化模型输出

- 保存最佳模型
- 保存最佳阈值
- 保存验证集和测试集指标
- 在报告中明确写出“默认阈值”与“最优阈值”对比，便于解释提升来源

### 2. 融合引擎优化

融合引擎当前的平均延迟是可接受的，但尾延迟不合格。结合现有脚本与报告特征，问题更可能来自请求链中的重复计算、模型加载、特征构造和序列化开销，而不是核心判别逻辑本身。

#### 融合路径的优化方向

- 预加载模型与必要资源，避免请求内重复加载
- 将固定或低频变化的特征预处理迁移到请求前或离线阶段
- 简化融合层输入，把结构化、文本、生理信号统一成低维概率或风险分，而不是在融合层做冗余的高维 one-hot / DataFrame 展开
- 给融合服务增加轻量级降级逻辑：当部分模态不可用或超时，使用已知稳定的轻量融合路径

#### 融合策略建议

优先采用“概率级融合”或“风险分级融合”，而不是再次构建复杂的高维联合特征。这样做的原因是：

- 结构化和文本模型已经分别给出了高质量的风险输出
- 融合层的职责应是整合信号，而不是重新做重训练
- 概率级融合更容易缓存、更容易解释，也更容易把延迟控制住

### 3. 评估与报告改进

为了避免只看单点结果，后续评估报告应明确区分：

- 主评估集：结构化与文本模型的大样本结果
- 小样本参考集：生理模型与融合测试场景
- 线上关注指标：平均延迟、`p95`、`p99`、`max`

报告中建议新增以下内容：

- 阈值优化前后对比
- 结构化模型各候选模型的验证集 F1 排名
- 融合引擎延迟分布的变化
- 若存在降级策略，需单独展示降级次数和降级后指标

## 详细设计

### A. 结构化模型优化设计

#### A.1 特征优化

当前脚本已包含：

- `Sleep Duration` 映射为序数特征
- `Dietary Habits` 映射为序数特征
- `Age` 分箱为 `AgeGroup`
- 若干压力、满意度和工时字段的数值化

这些方向是正确的，但还可以补强：

1. 对高缺失率字段进行更稳健的缺失指示
2. 对少量类别字段保留原始类别与序数派生特征的双通道表达
3. 对类别极不平衡字段做低频合并，减少噪声

#### A.2 模型选择

保留多模型搜索，但将筛选逻辑从“只看交叉验证默认 F1”扩展为：

- 交叉验证 F1
- 验证集 F1
- 验证集阈值优化后的 F1
- 推理复杂度

推荐优先级：

1. 结构简单、推理稳定的模型
2. 验证集 F1 可通过阈值提升的模型
3. 在测试集上泛化稳定的模型

#### A.3 阈值优化

这是本次结构化模型优化的关键点之一。

做法：

- 训练完成后，获取验证集预测概率
- 在 `0.1 ~ 0.9` 区间以固定步长扫描阈值
- 选择 F1 最优阈值
- 用该阈值重新评估测试集

这样做的好处是：

- 不增加在线模型复杂度
- 常常能明显改善 precision/recall 的平衡
- 对当前“略低于达标线”的场景非常有效

### B. 融合引擎优化设计

#### B.1 当前问题判断

当前融合引擎的 `avg_ms` 并不高，但 `p95` 明显超标，说明大部分请求正常，少数请求很慢。通常这意味着：

- 初始化或首次调用成本过高
- 数据展开、拼接、序列化造成偶发慢请求
- 某个模态不可用时触发了较重的回退逻辑

#### B.2 低延迟融合策略

建议采用以下顺序：

1. 预热模型和依赖
2. 缓存静态资源与常见中间结果
3. 融合层只消费低维风险分/概率
4. 避免在请求过程中做大规模 DataFrame 操作

#### B.3 降级策略

为融合引擎设计明确的降级条件：

- 文本或生理模态不可用
- 请求超时风险升高
- 某个上游模型暂时加载失败

降级时优先返回：

- 结构化模型结果
- 或结构化 + 文本的轻量组合结果

这样可以保证服务可用性，即使尾延迟偶尔抖动，也不至于影响整体交付。

## 预期收益

### 结构化模型

- 通过阈值优化和特征补强，F1 有较大概率跨过 `0.85`
- 模型解释性仍保持较强
- 不会显著增加推理成本

### 融合引擎

- 通过热路径减负和概率级融合，`p95` 延迟有望降到 `200 ms` 以内
- 极端慢请求会减少
- 降级策略可提升稳定性与可用性

### 系统整体

- 文本模型继续作为当前最稳的高性能子模型
- 结构化模型补齐短板
- 融合模块从“复杂但慢”转向“简单但稳定”

## 风险与应对

### 风险 1 结构化阈值优化提升有限

**应对**：继续从特征清理、类别合并、参数搜索空间扩展入手；必要时增加模型候选。

### 风险 2 融合尾延迟仍超标

**应对**：进一步减少在线特征构造，检查首次加载与缓存命中情况；必要时在融合层增加更激进的降级路径。

### 风险 3 小样本评估误导结论

**应对**：明确小样本结果只作为 sanity check，不作为主结论；后续补充更大的融合验证集。

### 风险 4 优化引入回退

**应对**：每一步修改后都保留基线模型和报告，确保可快速切回。

## 验收标准

优化完成后，需要满足以下条件：

1. 结构化模型测试集 F1 `>= 0.85`
2. 融合引擎 `p95` 延迟 `<= 200 ms`
3. 文本模型 F1 不低于当前水平的明显回退阈值
4. 评估报告可清楚展示优化前后变化
5. 训练和推理流程可以复现，且存在可回退的基线产物

## 推荐实施顺序

1. 为结构化模型增加阈值优化与更完整的验证集评估
2. 轻量扩展结构化特征处理与参数搜索
3. 将融合引擎改成低维输入与预加载模式
4. 增加融合降级策略和延迟指标观察
5. 重新生成性能报告并核对是否达标

## 结论

本次问题最优解不是大规模重构，而是“结构化模型做最小代价增益，融合引擎做热路径减负”。

该方案的优势在于：

- 能最快补上结构化模型 F1 的基准缺口
- 能最直接降低融合引擎尾延迟
- 对现有文本模型影响最小
- 风险可控，便于逐步验证和回退
