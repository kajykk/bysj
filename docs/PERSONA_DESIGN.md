# DWS 用户画像与 Persona 设计

> 基于 PRODUCT_EVALUATION_REPORT.md 改进项 I5，为抑郁症动态预警与干预系统（DWS）设计精细化用户画像，指导功能优先级决策与产品迭代方向。

---

## 1. Persona 概览

DWS 系统服务于三类核心角色，共设计 5 个 Persona，覆盖典型用户、边缘用户与决策者：

| # | Persona | 角色 | 年龄 | 技术熟练度 | 核心场景 |
|---|---------|------|------|-----------|---------|
| P1 | 林小雨 | 普通用户（大学生） | 20 | 中 | 日常风险评估与自助干预 |
| P2 | 张伟 | 普通用户（职场新人） | 25 | 中高 | 压力监测与隐私保护 |
| P3 | 陈思颖 | 心理咨询师 | 32 | 中 | 多用户管理与危机干预 |
| P4 | 王建国 | 系统管理员 | 38 | 高 | 系统运维与合规审计 |
| P5 | 刘阿姨 | 普通用户（中老年） | 55 | 低 | 简化操作与家庭支持 |

## 2. Persona 详细设计

### P1: 林小雨 — "关注自我心理健康的大学生"

**基本信息**
- 年龄：20 岁
- 身份：大二学生，心理学相关专业
- 技术熟练度：中（熟悉手机 App，较少使用 PC 端）
- 设备：iPhone 14 + iPad

**背景**
林小雨是心理学专业大二学生，对心理健康有基础认知。近期课业压力增大，偶尔出现失眠和焦虑情绪。学校推荐使用 DWS 系统进行自我监测。

**目标与动机**
- 了解自己的风险等级变化趋势
- 获取个性化的干预建议（冥想、运动等）
- 在感到压力时能快速完成评估

**痛点与需求**
- 不希望同学知道自己使用心理评估系统（隐私敏感）
- 评估问卷太长会放弃填写
- 希望看到可视化的趋势图，而非枯燥的数据

**使用场景**
1. 每周日晚在宿舍完成心理量表评估
2. 查看本周风险等级变化趋势图
3. 根据系统推荐进行 10 分钟冥想练习
4. 导出 PDF 报告提交给学校心理辅导中心

**用户旅程**
```
注册 → 首次评估（5min）→ 查看风险趋势 → 获取干预建议 → 4周后导出报告
```

**设计启示**
- 移动端优先，确保关键流程在手机上流畅
- 趋势图表需要清晰的可视化
- 评估流程支持断点续传
- 隐私保护策略需在注册时明确告知

### P2: 张伟 — "高压职场新人"

**基本信息**
- 年龄：25 岁
- 身份：互联网公司初级开发工程师
- 技术熟练度：中高（熟悉各类工具，关注数据安全）
- 设备：Android 手机 + ThinkPad

**背景**
张伟毕业一年，在互联网公司高强度工作。近期项目压力大，经常加班到深夜。公司 EAP 项目引入 DWS 系统供员工使用。

**目标与动机**
- 实时监测压力水平变化
- 在高风险时收到预警提醒
- 保护个人心理健康数据不被雇主查看

**痛点与需求**
- 担心评估数据被公司 HR 看到
- 希望通过生理数据（心率、睡眠）辅助判断
- 需要异步通知（不希望在工作时间被打扰）

**使用场景**
1. 早晨通勤时查看昨日风险评分
2. 连接智能手环同步生理数据
3. 高风险时收到 WebSocket 实时预警
4. 在午休时间查看干预建议

**用户旅程**
```
企业邀请注册 → 关联智能设备 → 日常被动监测 → 高风险预警 → 预约咨询师
```

**设计启示**
- 生理数据接入需支持主流可穿戴设备
- 预警通知需支持分级（静默/振动/弹窗）
- 隐私政策需明确"数据所有权属于用户"
- 异步任务进度反馈（I1 改进已实现）

---

### P3: 陈思颖 — "负责多用户管理的心理咨询师"

**基本信息**
- 年龄：32 岁
- 身份：高校心理咨询中心专职咨询师
- 技术熟练度：中（熟练使用办公软件，对数据图表理解有限）
- 设备：MacBook Air + iPhone

**背景**
陈思颖负责 200+ 名学生的心理辅导工作，每周需处理 15-20 个咨询案例。以前依赖纸质档案和 Excel 管理，效率低下。

**目标与动机**
- 快速识别高风险学生，优先介入
- 批量查看负责学生的风险趋势
- 记录咨询笔记并与评估数据关联

**痛点与需求**
- 学生太多，无法逐一查看，需要系统自动预警
- 需要按风险等级排序，聚焦高危学生
- 希望在系统中直接记录咨询笔记

**使用场景**
1. 每日早晨登录系统查看新增高风险预警
2. 按风险等级排序，逐一审核高危学生
3. 查看学生的历史评估数据与趋势图
4. 记录咨询笔记，更新干预状态
5. 每月导出统计报告提交给中心

**用户旅程**
```
登录 → 预警看板 → 高风险学生详情 → 咨询记录 → 干预跟踪 → 月度报告
```

**设计启示**
- 咨询师 Dashboard 需突出"今日高风险"卡片
- 支持批量操作（批量标记、批量导出）
- 咨询笔记需与时间线集成
- 列表页支持按多维度排序与筛选

### P4: 王建国 — "关注系统合规的管理员"

**基本信息**
- 年龄：38 岁
- 身份：高校信息化中心运维主管
- 技术熟练度：高（熟悉 Linux/Docker/监控系统）
- 设备：工作站 + 多显示器

**背景**
王建国负责学校多个信息系统的运维，DWS 是其中之一。他关注系统稳定性、数据安全和合规审计。

**目标与动机**
- 确保系统 7×24 稳定运行
- 监控异常指标（API 延迟、错误率、熔断状态）
- 满足等保 2.0 合规审计要求

**痛点与需求**
- 需要实时监控大盘，一屏掌握系统状态
- 操作日志需完整记录，支持追溯
- 告警需要分级通知（短信/邮件/Webhook）

**使用场景**
1. 每日检查系统监控大盘
2. 配置告警规则与静默策略
3. 审查操作日志，排查异常访问
4. 管理用户权限与角色分配
5. 执行系统健康检查与灰度发布

**用户旅程**
```
登录 → 监控大盘 → 告警处理 → 日志审计 → 权限管理 → 灰度发布
```

**设计启示**
- 管理员 Dashboard 需集成监控、告警、日志
- 支持可观测性指标（I1 改进的异步任务进度）
- 灰度发布与熔断状态需可视化
- 操作日志支持多维度筛选与导出

### P5: 刘阿姨 — "需要简化操作的中老年用户"

**基本信息**
- 年龄：55 岁
- 身份：退休教师，因家庭医生推荐使用
- 技术熟练度：低（仅会使用微信和短视频 App）
- 设备：华为手机（大字号模式）

**背景**
刘阿姨近期情绪低落，家庭医生建议使用 DWS 系统进行心理健康自测。她对智能手机操作不熟练，容易在复杂界面中迷失。

**目标与动机**
- 简单完成心理评估
- 获取易懂的结果解读
- 在需要时联系心理咨询师

**痛点与需求**
- 界面文字太小看不清
- 专业术语不理解（如"多模态融合""风险等级"）
- 操作步骤太多会放弃
- 希望有语音引导

**使用场景**
1. 在女儿帮助下完成注册
2. 每周完成一次简化版评估
3. 查看大字号的结果解读
4. 高风险时由系统自动联系咨询师

**用户旅程**
```
女儿协助注册 → 新手引导 → 简化评估 → 大字号结果 → 自动分配咨询师
```

**设计启示**
- 支持大字号/高对比度无障碍模式
- 评估问卷使用通俗语言，避免专业术语
- 新手引导系统（I2 改进已实现）需覆盖关键流程
- 关键操作不超过 3 步

## 4. 功能优先级矩阵

基于 Persona 分析，功能优先级如下：

### P0 — 核心路径（所有 Persona 必需）
- 风险评估与计算
- 风险趋势可视化
- 预警通知（WebSocket 实时推送）
- 异步任务进度反馈（I1 已实现）

### P1 — 高价值功能（覆盖 3+ Persona）
- 新手引导系统（I2 已实现）
- 帮助中心 FAQ（I7 已实现）
- 无障碍合规（I6 已实现，Accessibility 96→100）
- 咨询师批量管理工具
- 管理员监控大盘

### P2 — 增强功能（覆盖 2 Persona）
- 生理数据接入（智能手环）
- 大字号/高对比度模式
- 评估问卷断点续传
- 操作日志多维度筛选

### P3 — 长期规划（覆盖 1 Persona 或战略价值）
- 语音引导（刘阿姨）
- 用户社区与反馈渠道
- 多租户支持
- 商业模式设计

## 5. 设计决策指导

### 5.1 移动端优先 vs PC 端优先

| Persona | 主要设备 | 设计权重 |
|---------|---------|---------|
| 林小雨 | 手机 + iPad | 移动端优先 |
| 张伟 | 手机 + PC | 双端并重 |
| 陈思颖 | MacBook + iPhone | PC 端优先 |
| 王建国 | 工作站 | PC 端专属 |
| 刘阿姨 | 手机 | 移动端专属 |

**结论**：评估流程移动端优先，管理流程 PC 端优先。

### 5.2 信息密度策略

- **用户端**：低信息密度，大间距，聚焦当前任务
- **咨询师端**：中信息密度，列表+详情双栏布局
- **管理员端**：高信息密度，多面板仪表盘

### 5.3 无障碍优先级

基于 I6 修复（Accessibility 96→100）和刘阿姨 Persona：
- 色彩对比度 ≥ 4.5:1（WCAG AA）
- 支持键盘导航与屏幕阅读器
- 提供大字号模式（未来 P2 功能）
- 评估问卷使用通俗语言

## 6. 验证与迭代

### 6.1 验证方法
- **可用性测试**：邀请 5 名 Persona 代表用户完成核心任务
- **A/B 测试**：对比不同信息密度下的任务完成率
- **数据分析**：追踪各 Persona 的功能使用率与留存率

### 6.2 迭代周期
- Phase 1（已完成）：异步进度反馈 + 新手引导 + Lighthouse 实测
- Phase 2（进行中）：Persona 设计 + 无障碍修复 + 帮助系统 + 视觉令牌
- Phase 3（规划中）：用户反馈渠道 + 多租户 + 商业模式
