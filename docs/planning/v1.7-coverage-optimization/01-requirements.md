# v1.7 迭代需求文档

> **迭代名称**: v1.7-backend-contract-coverage-hardening
> **迭代目标**: 质量硬化 + 覆盖率提升 + 契约治理 + 前端工程规范补齐
> **上一迭代**: v1.6-contract-e2e-quality-governance (已完成)
> **建议周期**: 4 周
> **日期**: 2026-04-29

---

## 一、迭代背景

### 1.1 上一迭代遗留问题

v1.6 已完成测试基础设施、E2E 测试、覆盖率工具配置、质量工具链搭建，但最终报告显示仍存在以下关键问题：

| 问题 | v1.6 状态 | 风险等级 | v1.7 处理策略 |
|------|----------|---------|--------------|
| 后端覆盖率不足 | 36.29% | **P0** | 提升到 60%+，核心模块 75%+ |
| 后端测试仍有失败项 | 161 passed / 30 failed | **P0** | 先分类、修复或隔离 |
| 契约测试通过率不足 | 35.8% | **P0** | 提升到 80%+ |
| OpenAPI 缺少 401/403 | 不完整 | **P0** | 核心接口全部补齐 |
| ESLint / Prettier 缺失 | 未配置 | P1 | 建立可运行规范基线 |
| Chunk 体积偏大 | charts 812KB、vendor 620KB | P1/P2 | 先分析，再优化 |
| Sass Legacy API warning | 存在 | P2 | 溯源并记录升级路径 |

### 1.2 迭代动机

v1.6 完成了测试基础设施搭建，但工具链尚未落到稳定运行状态。v1.7 迭代专注于：

1. **先稳定，再扩展** — 修复现有失败测试，再补充新测试
2. **核心优先，边缘延后** — 优先覆盖 auth、user、prediction、services、ML 数据处理等高风险模块
3. **目标现实，分阶段达成** — v1.7 后端覆盖率目标设为 60%，不强行一次性达到 85%
4. **契约先补文档，再修实现** — 先解决 OpenAPI 缺少 401/403 等高收益问题
5. **前端规范先可运行，再逐步严格** — ESLint / Prettier 首轮以可执行为主
6. **性能优化先分析，再改造** — 对 chunk 体积先产出分析报告，再进行优化

> **核心定位**: v1.7 是一次 **质量硬化迭代**，不是新增大量业务能力，而是把 v1.6 建好的工具链真正落到稳定运行状态。

## 二、需求列表

### 2.1 P0 - 必达目标

#### R-P0-001: 后端失败测试收口
- **描述**: 收集当前失败测试，按原因分类，修复主路径失败项
- **验收标准**:
  - 主路径测试无阻塞失败
  - 所有失败项都有原因和处理结论
  - 产出 `TEST_FAILURE_ANALYSIS_V1.7.md`

#### R-P0-002: 后端覆盖率提升
- **描述**: 补充核心模块测试，提升后端整体覆盖率到 60%+
- **验收标准**:
  - 后端整体覆盖率 >= 60%
  - auth/user/prediction 覆盖率 >= 75%
  - services 核心逻辑覆盖率 >= 65%
  - 新增测试稳定通过

#### R-P0-003: OpenAPI 错误响应补齐
- **描述**: 为所有需要认证的端点添加 401/403 响应定义
- **验收标准**:
  - 所有 protected 端点定义 401 响应
  - 所有 admin 端点定义 403 响应
  - 核心接口 401/403 定义完整度 100%

#### R-P0-004: 契约测试通过率提升
- **描述**: 修复 Schemathesis 契约测试失败项
- **验收标准**:
  - schemathesis 通过率 >= 80%
  - 失败项均有分类说明
  - 产出 `CONTRACT_TEST_REPORT_V1.7.md`

#### R-P0-005: 前端构建持续健康
- **描述**: 确保前端 type-check 和 build 持续通过
- **验收标准**:
  - `npm run type-check` 通过
  - `npm run build` 通过
  - 前端生产构建持续可用

#### R-P0-006: pytest 覆盖率阈值分阶段调整
- **描述**: 当前 pytest.ini 设置 `--cov-fail-under=85`，与 v1.7 目标冲突，需分阶段调整
- **验收标准**:
  - Week 1: 移除 fail-under 或设为 35%（不低于当前 36.29%）
  - Week 2: 设为 50%
  - Week 3: 设为 55%
  - Week 4: 设为 60%
  - 核心模块 (auth/user/prediction) 单独检查 >= 75%

#### R-P0-007: v1.7 最终报告产出
- **描述**: 产出完整的 v1.7 迭代报告
- **验收标准**:
  - 所有关键指标有记录
  - 所有失败项有处理结论
  - v1.8 可继承

### 2.2 P1 - 应达目标

#### R-P1-001: ESLint / Prettier 配置
- **描述**: 配置前端代码规范工具
