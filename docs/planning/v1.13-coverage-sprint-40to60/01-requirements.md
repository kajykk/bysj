## 1. 项目概述 (Project Overview)

### 1.1 背景
v1.12 迭代已将后端测试覆盖率提升至 40%（目标），但仍有大量代码路径未覆盖。v1.13 迭代聚焦于将覆盖率从 40% 提升至 60%，重点覆盖 services/、api/、core/ 等关键模块。

### 1.2 目标
- **整体覆盖率**: 40% → 60%
- **services/ 覆盖率**: 目标 >= 50%
- **api/ 覆盖率**: 目标 >= 45%
- **core/ 覆盖率**: 目标 >= 55%
- **新增测试数**: 50+ 个

### 1.3 范围
- **In Scope**: backend/tests/ 新增测试，backend/app/ 中未覆盖模块
- **Out of Scope**: 前端测试、E2E 测试、性能测试

## 3. 非功能需求 (Non-Functional Requirements)

### 3.1 性能
- 测试执行时间: <= 5 分钟 (完整测试套件)
- 单个测试: <= 2 秒

### 3.2 质量
- 所有新增测试必须通过
- 不允许降低现有覆盖率
- 测试必须独立，不依赖外部服务

### 3.3 文档
- 每个测试文件必须包含模块级 docstring
- 复杂测试必须包含注释说明测试意图

## 5. 风险与假设

### 5.1 风险
- **环境限制**: 当前环境无法运行 pytest，需在 CI 或支持环境中验证
- **依赖复杂**: services 可能依赖数据库/外部 API，需要 Mock

### 5.2 假设
- 现有代码逻辑正确，测试仅验证行为
- 数据库模型已定义，可用于测试固件

## 6. 环境限制应对策略 (Round 2 补充)

### 6.1 本地开发环境
- 使用 SQLite 内存数据库替代 PostgreSQL
- 使用 fakeredis 替代 Redis
- 使用 unittest.mock 替代外部 HTTP 调用

### 6.2 CI 环境验证
- GitHub Actions 运行完整测试套件
- 覆盖率报告上传 Codecov
- 失败测试自动通知

### 6.3 降级策略
- 如果 pytest 无法运行，使用代码审查验证测试逻辑
- 如果 coverage 无法生成，使用估算方法评估覆盖率
- 记录所有环境限制和应对措施
