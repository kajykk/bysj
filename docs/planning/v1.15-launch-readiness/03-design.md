# v1.15 设计文档: 上线就绪与核心功能闭环

> **迭代名称**: v1.15-launch-readiness  
> **目标**: 以最短路径达到可上线、可验证、可回滚  
> **日期**: 2026-05-01  
> **状态**: Draft

---

## 1. 设计原则

v1.15 采用“上线优先”的设计原则：

1. **先跑通核心流程，再补齐非核心质量指标。**
2. **先确保目标环境可运行，再优化 Windows 本地测试体验。**
3. **先建立可执行清单，再做大规模自动化。**
4. **先消除 P0 阻塞，再处理 P1/P2 技术债。**
5. **不新增非必要功能，避免扩大上线风险。**

---

## 2. 功能闭环设计

### 2.1 核心流程定义

上线前必须明确并验证系统的核心用户流程。每条流程至少包含：

1. 用户入口页面。
2. 必要输入或操作。
3. 前端 API 请求。
4. 后端处理逻辑。
5. 数据库读写或模型/算法调用。
6. 前端结果展示。
7. 异常输入处理。

若某条流程无法完整跑通，必须记录到 `LAUNCH_BLOCKERS.md`。

### 2.2 核心流程验收方式

每条核心流程应使用以下方式验收：

- 手动执行一次成功路径。
- 手动执行一次关键失败路径。
- 截图或日志记录关键结果。
- 对应 API 使用接口工具或浏览器网络面板验证。
- 若可自动化，则补充为冒烟测试。

## 3. 部署闭环设计

### 3.1 前端部署设计

必须确认：

- 生产构建命令可运行。
- 构建产物可被静态服务或部署平台托管。
- API Base URL 通过环境变量配置。
- 路由刷新不出现 404 或白屏。

### 3.2 后端部署设计

必须确认：

- 后端启动命令明确。
- 依赖安装命令明确。
- 环境变量清单完整。
- 健康检查接口可用。
- CORS 白名单配置正确。
- 日志能输出启动失败原因。

### 3.3 数据库部署设计

必须确认：

- 数据库连接字符串配置明确。
- 初始化或迁移步骤明确。
- 目标环境有持久化策略。
- 核心表结构与代码匹配。

### 3.4 模型/算法部署设计

必须确认：

- 模型文件或算法依赖路径明确。
- 目标环境依赖可安装。
- Windows 专有问题不影响目标部署环境。
- 加载失败有错误记录。

---

## 4. 验证闭环设计

### 4.1 P0 验证

P0 验证只关注上线是否安全：

1. 前端构建通过。
2. 后端启动通过。
3. 健康检查通过。
4. 核心 API 通过。
5. 核心流程手动验收通过。
6. 回滚方案可执行。

### 4.2 P1 验证

P1 验证用于提升信心：

1. Docker/Linux 后端关键测试通过。
2. CI 前端构建通过。
3. CI 后端关键测试通过。
4. 覆盖率报告可生成。
5. 安全头、CORS、认证基础测试通过。

### 4.3 P2 验证

P2 验证用于上线后优化：

1. 覆盖率提升到 80%。
2. 完整 E2E 自动化。
3. Lighthouse 性能门禁。
4. 监控告警完善。

## 6. 交付决策设计

上线决策分为三种：

### 6.1 Go

满足所有 P0 标准，P1 无重大风险。

### 6.2 Conditional Go

P0 满足，但存在可接受的 P1 问题。必须在交付报告中记录风险、负责人和后续修复计划。

### 6.3 No-Go

任一 P0 标准未满足，禁止上线。
