# v1.15 架构文档: 上线就绪验证架构

> **迭代名称**: v1.15-launch-readiness  
> **目标**: 建立面向上线的功能闭环、部署闭环和验证闭环  
> **日期**: 2026-05-01  
> **状态**: Draft

---

## 1. 架构目标

v1.15 的架构关注点不是新增复杂能力，而是确认现有系统可以稳定上线。架构验证分为三条主线：

1. **业务功能闭环**: 用户从前端进入系统，完成核心业务操作，并看到正确结果。
2. **运行部署闭环**: 前端、后端、数据库、模型文件、环境变量在目标环境中可启动、可连接、可访问。
3. **质量验证闭环**: 用最小但稳定的 CI/Docker/冒烟测试证明系统没有上线阻塞问题。

## 2. 上线验证架构

```text
User Browser
    |
    | HTTP(S)
    v

    |
    | API Base URL / CORS / Auth
```

## 3. 核心组件职责

### 3.1 前端层

职责：

- 提供核心页面和用户交互入口。
- 调用后端 API。
- 展示成功结果、加载状态和错误提示。
- 支持生产构建和部署。

上线关注点：

- `npm run build` 或项目等价构建命令成功。
- API Base URL 配置正确。
- 核心页面无白屏。
- 核心流程刷新后仍可访问。
- 错误响应有可理解提示。

### 3.2 后端 API 层

职责：

- 提供健康检查、认证、核心业务 API。
- 处理请求校验、错误响应、CORS、安全头和限流。
- 调用数据库和业务/算法模块。

上线关注点：

- 服务可在目标环境启动。
- 健康检查接口可访问。
- 核心 API 返回符合前端预期的数据结构。
- 异常输入不会导致 500 或进程崩溃。

### 3.3 数据库层

职责：

- 保存系统核心业务数据。
- 支撑核心流程中的数据创建、读取、更新。

上线关注点：

- 数据库连接配置明确。
- 初始化或迁移步骤可执行。
- 核心数据读写验证通过。
- 连接失败时错误可定位。

### 3.4 模型/算法层

职责：

- 执行系统核心计算、预测或分析逻辑。
- 返回前端可展示的结果。

上线关注点：

- 模型依赖、模型路径、输入格式明确。
- 目标环境可以加载模型或执行降级逻辑。
- 典型输入可返回合理结果。
- 异常输入有明确错误响应。

### 3.5 CI/Docker 验证层

职责：

- 提供绕过 Windows DLL 问题的稳定验证环境。
- 运行上线前关键质量门禁。

上线关注点：

- 后端关键测试在 Docker/Linux/CI 中运行。
- 前端类型检查和构建在 CI 中运行。
- 覆盖率报告作为参考指标，不阻塞上线。

## 4. 环境策略

### 4.1 本地 Windows 环境

定位：开发调试环境。

限制：

- 不作为完整后端测试和覆盖率的唯一可信来源。
- 若 sklearn/pandas/numpy 触发 DLL 崩溃，允许跳过完整覆盖率验证。

### 4.2 Docker 或 Linux 环境

定位：后端测试与上线前验证主环境。

要求：

- 能安装后端依赖。
- 能启动后端服务。
- 能运行关键测试。
- 能生成覆盖率报告则生成，不能生成不阻塞 P0 上线。

### 4.3 生产/演示环境

定位：最终上线运行环境。

要求：

- 环境变量完整。
- 前端构建产物可访问。
- 后端服务可启动。
- 数据库连接正常。
- 健康检查可访问。

---

## 5. 错误处理策略

上线前必须覆盖以下错误处理：

1. API 地址配置错误时，前端应提示服务不可用。
2. 后端校验失败时，应返回结构化错误。
3. 数据库连接失败时，日志中应能定位配置问题。
4. 模型/算法加载失败时，应返回可理解错误或启用降级策略。
5. 未认证或权限不足时，应返回明确状态码。

---

## 6. 质量门禁架构

v1.15 质量门禁采用“最小但有效”的上线策略：

```text
Code Change
    |
    v
Frontend Type Check / Build
    |
    v
Backend Startup / Health Check
    |
    v
Core API Smoke Tests
    |
    v
Manual Core Flow Verification
    |
    v
Launch Decision
```

覆盖率、完整测试、性能测试作为增强项，不作为 P0 上线阻塞。

---

## 7. 风险与缓解

| 风险 | 影响 | 缓解策略 |
|---|---|---|
| Windows 本地 pytest 崩溃 | 无法本地完整验证 | 使用 Docker/Linux/CI 作为主验证环境 |
| 前后端接口不一致 | 核心流程失败 | 建立核心 API 验收清单 |
| 环境变量缺失 | 部署启动失败 | 输出 `.env.example` 与部署清单 |
| 模型依赖加载失败 | 核心结果不可用 | 验证模型路径、依赖与降级逻辑 |
| 覆盖率目标占用上线时间 | 延误上线 | 覆盖率降级为 P1/P2 支撑项 |
