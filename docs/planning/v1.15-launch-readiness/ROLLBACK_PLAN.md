# 回滚方案

> **迭代名称**: v1.15-launch-readiness  
> **用途**: 上线失败或出现严重问题时，快速恢复到可用状态  
> **状态**: Draft

---

## 1. 回滚目标

当上线后出现 P0 故障时，应优先恢复系统可用性，而不是现场修复复杂问题。

回滚目标：

- 恢复前端可访问。
- 恢复后端 API 可用。
- 恢复数据库数据一致性。
- 降低用户影响范围。
- 保留日志用于事后分析。

---

## 2. 回滚触发条件

出现以下任一情况，应立即评估回滚：

- 前端首页或核心页面无法访问。
- 后端服务无法启动或频繁崩溃。
- 健康检查失败且短时间无法恢复。
- 核心 API 大面积 5xx。
- 数据库连接失败。
- 核心模型/算法不可用。
- 出现数据破坏或数据丢失风险。
- 用户核心流程无法完成。

---

## 3. 回滚前检查

- [ ] 确认当前故障是否为 P0。
- [ ] 确认是否有快速修复方案。
- [ ] 确认上一稳定版本或备份可用。
- [ ] 保存当前错误日志。
- [ ] 保存当前部署版本号或提交号。
- [ ] 通知相关人员暂停继续发布。

---

## 4. 回滚步骤

### 4.1 前端回滚

- [ ] 切换到上一稳定前端构建产物。
- [ ] 清理或刷新静态资源缓存，如适用。
- [ ] 验证首页可访问。
- [ ] 验证核心页面可访问。

### 4.2 后端回滚

- [ ] 切换到上一稳定后端版本。
- [ ] 恢复上一稳定环境变量配置，如适用。
- [ ] 重启后端服务。
- [ ] 验证健康检查。
- [ ] 验证核心 API。

### 4.3 数据库回滚

如本次上线包含数据库变更，必须谨慎处理：

- [ ] 确认是否执行过 schema 变更。
- [ ] 确认是否需要恢复备份。
- [ ] 确认是否存在向后不兼容数据。
- [ ] 执行数据库回滚或补偿脚本前先备份当前数据。
- [ ] 验证核心数据读写。

### 4.4 模型/算法回滚

- [ ] 恢复上一稳定模型文件或算法配置。
- [ ] 验证典型输入返回结果。
- [ ] 验证异常输入不会崩溃。

---

## 5. 回滚后验证

- [ ] 前端首页可访问。
- [ ] 核心页面可访问。
- [ ] 后端健康检查通过。
- [ ] 核心 API 通过。
- [ ] 数据库读写正常。
- [ ] 核心流程手动验证通过。
- [ ] 错误率恢复正常。
- [ ] 记录回滚原因和后续修复计划。
