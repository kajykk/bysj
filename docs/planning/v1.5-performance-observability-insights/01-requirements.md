## 2. Epic 列表

### Epic A: 真实样本验证闭环 (P0)

**目标**: 验证 v1.4 模型在真实历史样本上的效果

**需求**:
- [ ] 构建真实样本验证集（覆盖正常、边界、异常、低质量样本）
- [ ] **验证集构建流程**:
  - 样本来源: 从生产环境近 6 个月的历史数据中分层抽样
  - 标注标准: 基于现有风险评估标准操作程序 (SOP)
  - 质量检查: 10% 样本双人标注，Kappa 系数 > 0.8
  - 样本量: 最低 500 条，建议 800-1000 条
- [ ] 定义离线评估指标（Accuracy、Precision、Recall、F1、AUC、MAE、RMSE）
- [ ] 对比旧版本与新版本输出差异
- [ ] 记录异常样本与失败原因（NaN/Inf/缺字段/空文本/极端值）
- [ ] 输出验证结论与是否可灰度建议

**验收标准**:
- 验证集 ≥ 500 条样本，覆盖所有风险等级
- 指标对比表完整，包含提升样本与退化样本分析
- 异常样本清单明确，失败原因可定位

### 7.1 灰度流量分配算法 (对应 Critical Issue #3)

**调研发现**:
- **MLRun / Google Cloud**: 业界普遍采用 `hash(user_id) % 100 < traffic_percent` 的稳定哈希算法，确保同一用户始终路由到同一版本
- **最佳实践**: 
  - 使用 **稳定哈希 (Stable Hash)** 而非随机分配，保证用户体验一致性
  - 流量比例配置应支持 **动态更新** (通过 Redis/配置中心)，无需重启服务
  - 初始灰度比例建议 **1-5%**，而非直接 5%，进一步降低风险
- **推荐方案**: 
  - 算法: `md5(user_id)[0:8] % 100 < traffic_percent`
  - 动态配置: Redis 存储当前灰度比例，读取延迟 < 10ms
  - 初始比例: 1% -> 5% -> 25% -> 50% -> 100%

---

### 7.2 监控数据存储策略 (对应 Critical Issue #2)

**调研发现**:
- **TimescaleDB vs PostgreSQL**: 
  - TimescaleDB 作为 PostgreSQL 扩展，在 1 亿行数据量下写入吞吐量是 PostgreSQL 的 **20 倍** (111K vs 5K rows/s)
  - 范围聚合查询 (1h) TimescaleDB 仅需 3 分钟，PostgreSQL 需要 28 分钟
  - TimescaleDB 2.21 支持 Direct to Columnstore，峰值可达 **100M records/s**
- **MLOps 监控标准**: Prometheus + Grafana 是机器学习监控的事实标准，支持 `/metrics` 端点暴露自定义指标
- **推荐方案**:
  - **Phase 1 (v1.5)**: PostgreSQL + 异步批处理写入 ( Celery 任务每 30s 批量 flush )
  - **Phase 2 (v1.6+)**: 评估迁移至 TimescaleDB，利用 hypertable 自动分区
  - **指标采集**: 使用 Prometheus 采集服务指标，Grafana 可视化 (可选)
  - **数据保留**: 监控日志保留 90 天，聚合后数据保留 1 年

---

### 7.3 ECharts 包体积优化 (对应 Critical Issue #5 / Major Issue #9)

**调研发现**:
- **ECharts 5 按需加载**: 通过 `echarts/core` + 按需引入图表类型，可将包体积从 **~800KB** 降至 **~150KB**
- **Vue 3 官方推荐**: 使用 ESM 动态导入 (`import()`) 实现代码分割，配合 Vite 自动 chunk
- **按需引入示例**:
  ```js
  import { init, use } from 'echarts/core';
  import { CanvasRenderer } from 'echarts/renderers';
  import { LineChart, BarChart } from 'echarts/charts';
  import { GridComponent, TooltipComponent } from 'echarts/components';
  ```
- **推荐方案**:
  - 采用 **按需引入**，仅加载 LineChart/BarChart/PieChart + 必要组件
  - 预期包体积增加 **< 200KB** (gzip 后 < 60KB)
  - 不影响 "初始打包体积减少 20%" 的目标 (通过路由懒加载抵消)

### 7.4 PDF 生成方案 (对应 Critical Issue #5)

**调研发现**:
- **前端方案 (html2canvas + jsPDF)**: 
  - 问题: 阻塞主线程，大 DOM 时克隆步骤耗时 5-10s，总导出时间 8-12s
  - 用户体验: 导出期间页面无响应
- **后端方案 (ReportLab / WeasyPrint)**:
  - 优势: 不阻塞前端，服务端渲染稳定，总时间 4-6s
  - 劣势: 增加服务器负载
- **推荐方案**:
  - **主方案**: 后端 ReportLab 生成 PDF，通过流式响应返回
  - **备选方案**: 前端 html2canvas + jsPDF (后端不可用时降级)
  - **优化**: 大报告使用 Celery 异步生成，生成完成后通知用户下载

---

### 7.5 MLOps 监控告警最佳实践 (对应 Major Issue #6)

**调研发现**:
- **Prometheus + Grafana 标准架构**: 
  - 推理服务暴露 `/metrics` 端点
  - Prometheus 定时拉取 (pull-based)
  - Grafana 可视化 + 告警规则

> **状态**: 调研完成，针对 Round 2 Critique 新增的 4 个问题 (#15-18) 和待修复的 6 个问题 (#1, #7, #8, #10, #11, #13, #14) 进行外部调研

### 10.1 Celery Beat 高可用方案 (对应 Round 1 Issue #14)

**调研发现**:
- **Celery-BeatX**: 通过 Redis 分布式锁实现多 beat 实例的 active-passive 模式，只有持有锁的实例调度任务，锁过期后备用实例自动接管
- **RedBeat**: 同样使用 Redis 分布式锁，支持配置 `redbeat_lock_timeout` 控制故障转移时间（默认 5 分钟）
- **Celery-Redundant-Scheduler**: 提供同步调度器类，支持故障转移，默认 Redis 后端
- **推荐方案**: 使用 **RedBeat** 替代默认 Celery beat，配置 `redbeat_lock_timeout=300`（5 分钟故障转移），部署 2 个 beat 实例（1 主 1 备）

### 10.2 Vue 3 + Element Plus 国际化 (对应 Round 1 Issue #13)

**调研发现**:
- **vue-i18n**: Vue 3 官方国际化方案，支持 `createI18n` 配置 locale/fallbackLocale/messages
- **Element Plus i18n**: 通过 `ConfigProvider` 或全局配置 `locale` 实现组件语言切换，支持 zh-cn/en 等 60+ 语言
- **实现成本**: 监控面板和报告中心的新页面需要补充翻译键值，预估 1-2 天工作量
- **推荐方案**: v1.5 中监控面板和报告中心支持 i18n（延续 v1.3 的 vue-i18n 方案），但翻译内容可延后到 v1.5.1 补充

### 10.3 验证集构建最佳实践 (对应 Round 1 Issue #8)

**调研发现**:
- **分层抽样 (Stratified Sampling)**: 确保验证集覆盖所有风险等级和关键人口统计变量
- **时间序列验证**: 若数据有时间属性，验证集应在训练集之后，模拟真实预测场景
- **标注指南**: 必须有清晰、文档化的标注指南，确保标签一致性
- **质量检查**: 建议双人标注 + 仲裁机制，Kappa 系数 > 0.8 视为一致
- **推荐方案**:
  - 样本来源: 从生产环境近 6 个月的历史数据中随机抽样
  - 标注标准: 基于现有风险评估标准操作程序 (SOP)
  - 质量检查: 10% 样本双人标注，Kappa > 0.8
  - 样本量: 500 条为最低要求，建议 800-1000 条以提高统计显著性

### 10.4 监控下钻 API 设计参考 (对应 Round 2 Issue #16)

**调研发现**:
- **Datadog / Grafana**: 下钻能力通常通过 "从聚合指标跳转到关联日志/追踪" 实现
- **API 设计模式**: `GET /metrics/{metric_name}/details?time_range=...&filters=...`
- **推荐方案**: 
  - 接口: `GET /monitoring/request-details?alert_id={id}&limit=50`
  - 返回: 关联的请求列表（user_id, timestamp, model_version, input_summary, output_summary, latency_ms）
  - 权限: Admin  only

### 10.5 异步任务通知机制 (对应 Round 2 Issue #17)

**调研发现**:
- **标准模式**: 提交异步任务后返回 `task_id`，客户端轮询 `GET /tasks/{task_id}/status`
- **通知渠道**: 任务完成后通过 WebSocket 推送或站内信通知
- **推荐方案**:
  - 轮询接口: `GET /api/v1/tasks/{task_id}/status` (返回 PENDING/RUNNING/SUCCESS/FAILED)
  - 通知: 任务完成后发送站内信通知（含下载链接）
  - 下载: `GET /api/v1/tasks/{task_id}/download`（有效期 24 小时）

### 10.6 灰度回滚通知模板 (对应 Round 2 Issue #18)

**推荐模板字段**:
- 标题: "[回滚告警] 模型 {version} 已自动回滚至 {baseline_version}"
- 内容: 回滚原因 / 触发时间 / 影响用户数 / 当前状态 / 建议操作 / 查看详情链接

### 13.1 RedBeat 高可用配置验证

**调研发现**:
- RedBeat 使用 Redis 分布式锁确保只有一个 beat 实例运行
- 锁默认超时: `beat_max_loop_interval * 5` = 1500 秒 (25 分钟)
- 推荐配置: `REDBEAT_LOCK_TIMEOUT = 300` (5 分钟故障转移)
- 锁在超时时间的一半时自动续期 (心跳机制)
- 备用实例在锁释放后自动获取并接管

**推荐配置**:
```python
# celeryconfig.py
REDBEAT_REDIS_URL = 'redis://localhost:6379/1'  # 使用独立 Redis DB
REDBEAT_KEY_PREFIX = 'v1.5:celery:beat:'
REDBEAT_LOCK_KEY = 'v1.5:celery:beat:lock'
REDBEAT_LOCK_TIMEOUT = 300  # 5 分钟故障转移
CELERYBEAT_MAX_LOOP_INTERVAL = 5  # 更快的 beat 循环
```

**启动命令**:
```bash
celery beat -S redbeat.RedBeatScheduler
```

### 13.2 ECharts 按需引入配置验证

**调研发现**:
- ECharts 5.x 提供模块化导出结构
- 按需引入可将包体积从 ~600KB 压缩至 ~150KB
- 需要显式注册: `echarts.use([CanvasRenderer, LineChart, GridComponent, ...])`

**推荐配置**:
```typescript
// composables/useEcharts.ts
import * as echarts from 'echarts/core'
import { CanvasRenderer } from 'echarts/renderers'
import { LineChart, BarChart, PieChart } from 'echarts/charts'
import {
  GridComponent, TooltipComponent, LegendComponent,
  TitleComponent, DataZoomComponent
} from 'echarts/components'

// 注册必要模块
const registered = ref(false)
export function useEcharts() {
  if (!registered.value) {
    echarts.use([
      CanvasRenderer, LineChart, BarChart, PieChart,
      GridComponent, TooltipComponent, LegendComponent,
      TitleComponent, DataZoomComponent
    ])
    registered.value = true
  }
  return echarts
}
```

**注意事项**:
- 图表容器必须设置明确高度 (CSS height)
- 组件卸载时必须调用 `chart.dispose()` 防止内存泄漏
- 使用 ResizeObserver 替代 window.resize 监听

### 13.3 调研结论

| 方案 | 可行性 | 关键配置 |
|------|--------|----------|
| RedBeat 高可用 | ✅ 可行 | `REDBEAT_LOCK_TIMEOUT = 300`，部署 2 实例 |
| ECharts 按需引入 | ✅ 可行 | 显式 use() 注册，体积 < 150KB |
