# ADR-001: 多租户架构决策

> **状态**：已接受
> **日期**：2026-07-12
> **决策者**：DWS 开发团队
> **关联**：Phase 5 多租户与商业化验证

---

## 一、背景

DWS 心理健康风险评估系统计划从单校试点扩展到多校部署。Phase 5 要求实施"最小多租户切片"，支持：
- `tenant_id` 请求上下文
- RBAC 绑定
- 数据查询强制隔离
- 租户配置/品牌
- 租户级审计与导出
- 自动化越权测试防串租

---

## 二、决策选项分析

### 选项 A：独立部署（每个租户独立实例）

**优点**：
- 物理隔离，安全性最高
- 定制化能力强
- 故障隔离

**缺点**：
- 运维成本高（N 个实例 = N 倍运维）
- 功能同步困难
- 不适合 SaaS 模式

### 选项 B：共享数据库，独立 Schema

**优点**：
- 逻辑隔离较好
- 每个租户独立 schema，数据分离清晰

**缺点**：
- Schema 迁移需对所有租户执行
- 连接池管理复杂
- 租户数量增长时管理困难

### 选项 C：共享数据库，共享 Schema（多租户字段）✅ 选择

**优点**：
- 运维成本最低
- 功能同步简单
- 适合 SaaS 模式
- 水平扩展容易

**缺点**：
- 隔离依赖应用层（需严格 tenant_id 过滤）
- 误操作风险（漏加 tenant_id 条件）
- 需要自动化越权测试保障

**风险缓解**：
- 所有租户敏感查询必须通过 `tenant_scoped_query` 工具函数
- 自动化越权测试覆盖所有租户敏感端点
- 数据库约束 + 应用层双重保护

### 选项 D：混合模式（核心共享 + 高隔离独立）

**优点**：
- 灵活满足不同隔离需求

**缺点**：
- 架构复杂度高
- 两套代码路径维护成本高

---

## 三、最终决策

**采用选项 C：共享数据库，共享 Schema（多租户字段）**

理由：
1. Phase 5 计划明确"默认优先共享数据库共享 schema"
2. 当前处于产品化早期，租户数量少，运维成本是关键
3. 通过 `tenant_id` 字段 + 应用层强制过滤实现隔离
4. 对高隔离要求客户可保留独立部署方案（选项 A）作为降级

---

## 四、实施方案

### 4.1 数据模型变更

新增 `tenants` 表：
```sql
CREATE TABLE tenants (
    id INTEGER PRIMARY KEY,
    name VARCHAR(100) NOT NULL,          -- 租户名称（如"XX大学"）
    code VARCHAR(50) UNIQUE NOT NULL,    -- 租户编码（如"xx_univ"）
    status VARCHAR(20) DEFAULT 'active', -- active/inactive/suspended
    config JSON,                         -- 租户配置（品牌、功能开关等）
    created_at DATETIME,
    updated_at DATETIME
);
```

在 `users` 表添加 `tenant_id` 字段：
```sql
ALTER TABLE users ADD COLUMN tenant_id INTEGER REFERENCES tenants(id);
```

### 4.2 请求上下文

- 中间件从请求头 `X-Tenant-ID` 或子域名解析租户
- 租户上下文存储在 `request.state.tenant_id`
- 依赖注入 `get_current_tenant` 提供给路由使用

### 4.3 查询隔离

- 所有租户敏感表的查询必须包含 `WHERE tenant_id = :current_tenant_id`
- 提供 `tenant_scoped_query(model, tenant_id)` 工具函数
- 自动化越权测试验证跨租户访问被阻断

### 4.4 RBAC 绑定

- 用户角色绑定到租户（`user.tenant_id`）
- 管理员只能管理本租户用户
- 跨租户操作需要超级管理员权限

### 4.5 审计隔离

- `OperationLog` 添加 `tenant_id` 字段
- 审计日志查询自动过滤当前租户
- 超级管理员可查询所有租户审计日志

---

## 五、向后兼容性

- 现有单租户数据 `tenant_id = NULL` 或 `tenant_id = 1`（默认租户）
- 未认证租户的请求路由到默认租户
- 现有 API 在添加 `tenant_id` 后仍可正常工作

---

## 六、风险与缓解

| 风险 | 概率 | 影响 | 缓解措施 |
|------|------|------|---------|
| 漏加 tenant_id 导致数据泄露 | 中 | 严重 | 强制使用 `tenant_scoped_query` + 自动化越权测试 |
| 租户数据迁移错误 | 低 | 高 | 迁移脚本 + 数据校验 + 回滚方案 |
| 性能退化（tenant_id 索引） | 低 | 中 | 添加复合索引 + 性能基准测试 |
| 跨租户权限提升 | 低 | 严重 | RBAC 绑定 tenant_id + 权限测试 |

---

## 七、验证标准

| # | 验证项 | 目标 | 方式 |
|---|--------|------|------|
| V1 | 租户隔离 | 100% 阻断跨租户访问 | 自动化越权测试 |
| V2 | 查询性能 | P95 < 100ms | 性能基准测试 |
| V3 | 审计隔离 | 审计日志按租户隔离 | 集成测试 |
| V4 | 向后兼容 | 现有功能不受影响 | 回归测试 |

---

## 八、后续演进

- 如果租户数量 >50，考虑分库或分 schema
- 如果高隔离客户出现，支持独立部署模式
- 如果需要租户级计费，在 `tenants.config` 中扩展
