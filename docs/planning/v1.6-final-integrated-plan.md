# v1.6 稳定性治理与质量闭环收口版（最终整合方案）

> **版本名称**：v1.6-final-integrated-plan
>
> **版本定位**：稳定性治理与质量闭环收口版
>
> **核心目标**：在 v1.4 已完成模型链路修复与回退机制建设、v1.5 已完成工程化安全底座与前端体验/可视化增强的基础上，进一步通过稳定性治理、契约测试、E2E 测试、覆盖率提升与质量工具链集成，将系统推进到 **长期稳定、契约可信、流程可验、质量可守、体验可交付** 的生产化阶段。
>
> **适用范围**：项目评审、研发排期、测试验收、上线治理、后续维护

## 1. 背景与现状

结合 `RALPH_STATE.md`、`FULL_TEST_RUN_REPORT.md` 以及 `v1.6-contract-e2e-quality-governance` 方案，可以确认当前项目已进入一个非常关键的成熟阶段：

### 1.1 已完成的基础能力

当前系统已经具备以下稳定基础：

- 后端模型链路、输入校验、回退机制、漂移检测、报告导出、可选依赖兼容等能力已通过全量测试
- 前端测试环境、骨架屏、图表交互、监控面板、路由懒加载等能力已通过全量测试
- 全量测试规模达到：
  - 后端 `777 passed, 2 skipped`
  - 前端 `515 passed`
  - lint 无错误

这说明系统已经不是“功能能不能做出来”的问题，而是进入了 **长期稳定性治理和质量收口** 的阶段。

### 1.2 当前仍存在的关键问题

测试报告中仍保留若干 warning 和版本风险，说明系统虽然已稳定可用，但仍存在长期维护隐患：

- `sklearn` 模型反序列化版本不一致 warning
- `datetime.utcnow()` deprecation warning
- `sklearn imputer` 对全空特征的 warning
- 可选依赖在不同环境下的行为一致性仍需治理
- 测试通过后仍需要固化契约、E2E 和覆盖率基线，防止后续迭代引入回归

### 1.3 版本演进判断

从版本演进角度看：

- **v1.4**：修复模型链路与稳定性基础
- **v1.5**：完成工程化安全底座、监控与前端体验增强
- **v1.6**：应当完成稳定性治理、质量体系收口与体验闭环固化

因此，v1.6 不适合继续做大规模功能扩张，而应作为 **系统质量和长期维护的收官版本**。

## 2. 版本目标

v1.6 的总体目标可以概括为一句话：

> **把系统从“已经能用、能测、能看”进一步推进到“长期稳定、契约可信、流程可验、质量可守、体验可交付”的成熟生产化状态。**

具体目标如下：

1. **清理已知技术债**
   - 处理 warning、deprecated API、版本不一致、边界异常等问题。

2. **固化系统稳定性**
   - 强化模型推理、输入校验、回退机制、报告导出和监控链路的确定性。

3. **建立质量治理体系**
   - 通过契约测试、E2E 测试、覆盖率提升、CI/CD 门禁构建持续质量保障。

4. **完成监控与可观测性收口**
   - 让异常可发现、可定位、可追踪、可分级处理。

5. **完成体验闭环**
   - 把前端性能、图表展示、报告导出和移动端基础体验打磨到稳定可交付水平。

## 3. 设计原则

### 3.1 稳定优先

v1.6 的第一原则不是增加新模块，而是减少不确定性。

### 3.2 版本一致

训练、推理、测试、部署环境必须尽量保持一致，至少保证兼容策略明确且可追踪。

### 3.3 边界收口

对于空值、全空列、极端值、缺字段、非法类型、不可用依赖等情况，必须统一处理口径。

### 3.4 可观测、可回滚

任何异常都必须能够被记录、定位，并且在必要时快速回退到稳定版本。

### 3.5 测试驱动收口

凡是被修复的问题，都必须有对应测试覆盖或回归基线。

### 3.6 质量前置

契约测试、E2E 测试和覆盖率门禁必须前置于持续集成流程，而不是只在版本末期补测。

## 4. 综合方案总览

本版本采用 **“三层主线 + 一层收口”** 的结构：

### 第一层：稳定性底座
处理 warning、依赖差异、deprecated API、边界异常与 fallback 收口。

### 第二层：质量验证体系
通过契约测试、E2E、覆盖率和 CI 门禁保障系统不被改坏。

### 第三层：体验与交付闭环
将前端性能、图表展示、监控面板和报告导出收口为稳定交付件。

### 第四层：技术债收口
通过 warning 清理、类型安全提升、测试基线固化和文档沉淀降低后续维护成本。

## 6. 详细方案设计

---

### Epic A：版本一致性与依赖治理

#### 6.1 目标

消除当前测试报告中明确指出的 warning 和版本风险，确保训练、测试、推理、部署环境的一致性或兼容性明确。

#### 6.2 重点问题

- `sklearn` 模型反序列化版本不一致 warning
- `datetime.utcnow()` deprecation warning
- `sklearn imputer` 对全空特征的 warning
- PyTorch / Transformers 可选依赖在不同环境中的一致性

#### 6.3 具体任务

1. **统一 sklearn 版本策略**
   - 固定训练环境与推理环境的版本范围
   - 对跨版本加载模型明确支持范围
   - 必要时重新导出模型或增加兼容层

2. **替换 deprecated 时间 API**
   - 用现代 UTC 时间写法替换 `datetime.utcnow()`
   - 统一项目内时间处理风格

3. **治理全空特征 warning**
   - 在预处理阶段对全空列进行明确处理
   - 统一填充、过滤或告警策略
   - 避免 warning 成为隐藏异常来源

4. **整理可选依赖策略**
   - 明确 PyTorch、Transformers 缺失时的降级行为
   - 保证 fallback 机制可预测、可测试

#### 6.4 交付物

- 依赖锁定说明
- 环境兼容矩阵
- warning 清理记录
- 模型导出/加载规范

- 已知 warning 清理完毕或有明确豁免
- 训练、测试、推理环境一致性可解释、可验证
- 边界条件处理统一

### 9.4 可观测性验收

- 模型、数据、服务异常都能定位和告警
- 回退链路可快速触发
- 告警策略已降噪并分级

### 9.5 体验闭环验收

- 前端性能和骨架屏体验稳定
- 图表和 Dashboard 表达清晰
- 报告导出可稳定使用
- 核心页面基础移动端体验可用

---

## 10. 风险与应对

### 风险 1：warning 治理带来兼容性回归

**应对**
- 每项修改后立即执行对应测试
- 保留旧行为作为回退参考
- 必要时引入兼容层而非直接破坏性替换

### 风险 2：依赖版本统一影响当前测试稳定性

**应对**
- 先锁定可验证版本范围
- 再逐步收敛到最终版本
- 不一次性大面积升级

### 风险 3：边界收口过度导致业务灵活性下降

**应对**
- 区分硬错误和软警告
- 明确哪些输入应该被拒绝，哪些应该降级处理

### 风险 4：测试与门禁范围过大

**应对**
- 先完成 P0、P1 核心能力
- 非核心能力延期到后续版本
- 保证主链路不因质量优化而引入不稳定

### 风险 5：前端体验收口范围过大

**应对**
- 先交付核心页面
- 非核心页面延期到后续版本
- 优先保证主链路稳定

## 13. 总结

v1.6 的核心任务不是继续扩张功能边界，而是完成系统的稳定性治理、质量体系建设和体验闭环收口。

它应该承担的角色是：

- 把 warning 变成明确可控的技术债
- 把边界条件变成统一规范
- 把测试通过变成长期基线
- 把契约/E2E/覆盖率门禁变成真正可执行的质量防线
- 把监控和回退变成真正可运维的能力
- 把前端体验和报告能力收口为稳定交付件

最终，v1.6 将成为项目从“成熟可用”走向“长期可维护”的关键版本。
