# Phase 3：受控试点与临床/模型验证

> 本文档为 DWS 心理健康风险评估系统 12 个月产品化计划的 Phase 3 总览，覆盖第 5–8 月的"受控试点与临床/模型验证"阶段。Phase 3 在 Phase 2 研究准备之上，执行受控真实试点，完成行为、模型、公平性和安全验证。

| 基础设施 | Commit | 说明 |

### 1.1 模型验证基础设施

- **核心指标**：sensitivity、specificity、PPV、NPV、AUROC、Brier Score
- **置信区间**：Wilson score（比例指标）+ Bootstrap AUC（1000 次重采样）
- **多类指标**：按风险层级 one-vs-rest 计算
- **公平性检查**：按群体分组，`min_group_size=30` 小样本保护
- **API**：`POST /api/v1/validation/clinical`（仅管理员）

### 1.2 模型预测暂停开关

- **功能**：危机事件/重大误判时立即停止所有模型预测端点
- **检查点**：tabular/text/physiological/fusion 4 个预测端点
- **状态存储**：Redis 优先 + 内存降级，5s 本地缓存
- **API**：`POST /api/v1/model-kill-switch/activate|deactivate`、`GET /status`（仅管理员）
- **审计**：所有激活/解除操作记录到 `OperationLog`

### 2.3 模型门槛

试点正式启用前，模型必须达到以下最低性能指标（基于内部验证集）：

| # | 指标 | 目标值 | 风险层级 | 验证方式 |
|---|------|--------|---------|---------|
| M1 | Sensitivity（灵敏度） | ≥ 0.70 | 高风险（≥60 分） | `model_validation.py` |
| M2 | Specificity（特异度） | ≥ 0.75 | 高风险（≥60 分） | `model_validation.py` |
| M3 | PPV（阳性预测值） | ≥ 0.60 | 高风险（≥60 分） | `model_validation.py` |
| M4 | NPV（阴性预测值） | ≥ 0.90 | 高风险（≥60 分） | `model_validation.py` |
| M5 | AUROC | ≥ 0.75 | 二分类 | `model_validation.py` |
| M6 | Brier Score | ≤ 0.20 | 二分类 | `model_validation.py` |
| M7 | 公平性差异 | ≤ 0.15 | sensitivity gap | `model_validation.py` |

> 注：以上指标基于"高风险"层级（≥60 分），即模型识别真正需要关注的学生 vs 不需要关注的学生。Sensitivity ≥ 0.70 意味着至少 70% 真正高风险学生会被识别。

### 2.4 暂停门槛（自动触发暂停的条件）

以下任一条件触发时，必须立即激活暂停开关：

| # | 触发条件 | 响应动作 | 责任人 |
|---|---------|---------|--------|
| P1 | 重大误判（漏报高风险导致危机事件） | 立即暂停 + 24h 内复盘 | 管理员 + 心理中心 |
| P2 | 模型指标严重退化（Sensitivity < 0.50） | 立即暂停 + 排查原因 | 管理员 |
| P3 | 公平性差异扩大（sensitivity gap > 0.30） | 立即暂停 + 公平性审计 | 管理员 |
| P4 | 隐私/安全事件 | 立即暂停 + 安全响应 | 管理员 + 安全负责人 |
| P5 | 用户投诉率异常（>5% 用户投诉误判） | 评估后决定是否暂停 | 管理员 |
| P6 | 数据质量问题（输入数据分布显著漂移） | 评估后决定是否暂停 | 管理员 |

## 三、回滚条件与流程

### 3.1 回滚条件

| # | 回滚场景 | 触发条件 | 回滚动作 |
|---|---------|---------|---------|
| R1 | 模型版本回滚 | 新版本指标低于旧版本 | 切换回旧版本模型文件 + 审计日志 |
| R2 | 阈值回滚 | 阈值调整后误报率 >30% 或漏报率 >20% | 恢复旧阈值配置 + 审计日志 |
| R3 | 全局暂停 | 暂停门槛触发 | 激活 kill switch + 人工接管 |
| R4 | 试点终止 | 连续 2 次暂停事件或伦理违规 | 终止试点 + 数据封存 + 复盘报告 |

### 3.2 回滚流程

```
[触发条件检测]
       |
       v
[激活暂停开关] --> [通知管理员 + 心理中心]
       |
       v
[影响评估] --> [受影响用户清单 + 事件等级]
       |
       v
[执行回滚] --> [模型版本/阈值/全局暂停]
       |
       v
[审计记录] --> [OperationLog: operator, action, reason, timestamp]
       |
       v
[复盘报告] --> [根因分析 + 改进措施 + 重新上线审批]
```

### 3.3 人工覆盖机制

- **原则**：模型输出仅作为辅助参考，不自动触发惩罚性或医疗决定
- **高风险结果**：必须进入人工复核流程（ReviewService）
- **危机事件**：立即升级到心理中心（CrisisEventService）
- **人工覆盖记录**：所有人工覆盖操作记录到 `OperationLog`，包含原始结果、覆盖后结果、操作人、原因

## 四、分批启用计划

### 4.1 阶段划分

| 阶段 | 持续时间 | 参与人数 | 范围 | 门禁 |
|------|---------|---------|------|------|
| S0 内部演练 | 1 周 | 5-10 人（开发团队） | 全功能测试 | E1-E5 全部通过 |
| S1 小样本观察 | 2 周 | 20-30 人（志愿者） | 真实数据采集 | S0 无重大问题 |
| S2 扩大试点 | 4 周 | 50-100 人 | 行为指标采集 | S1 指标达标 |
| S3 持续运行 | ≥4 周 | ≥100 人 | 长期稳定性 | S2 指标达标 |

### 4.2 S0 → S1 升级条件

- 所有预测端点功能正常
- 暂停开关可用且已测试
- 审计日志可追溯
- 数据导出/删除功能正常（GDPR 合规）

### 4.3 S1 → S2 升级条件

- 无重大误判事件
- 用户任务成功率 ≥ 85%
- 模型指标达到 M1-M7 门槛
- 无未关闭的隐私/安全事件

### 4.4 S2 → S3 升级条件

- 连续 30 天行为数据采集完成
- 模型指标稳定（无连续下降趋势）
- 公平性检查通过
- 心理中心确认复核流程顺畅

## 五、试点评审机制

### 5.1 定期评审

- **频率**：每两周一次
- **参与方**：开发团队、心理中心、校方代表
- **内容**：模型指标、行为指标、安全事件、用户反馈、改进项
- **输出**：评审纪要、改进 backlog、Go/No-Go 决策

### 5.2 紧急评审

- **触发**：暂停开关激活、危机事件、安全事件
- **时限**：事件发生后 24 小时内
- **内容**：根因分析、影响评估、改进措施、是否恢复试点
- **输出**：事件复盘报告
