# v1.6 稳定性治理与质量闭环收口版 - 可执行任务清单

> **来源**：`v1.6-final-integrated-plan.md`
>
> **目标**：将最终方案拆解为可执行、可验收、可排期的原子任务清单。
>
> **执行原则**：按依赖顺序推进，优先完成 P0，再推进 P1，最后收口 P2。

---

## 0. 执行总则

### 0.1 任务拆解原则

- 任务必须足够原子化，单个任务建议控制在 1-4 小时内可完成
- 每个任务都必须有明确的输入、输出和验证方式
- 任务之间必须遵循依赖顺序：**稳定性治理 → 契约/E2E → 覆盖率/门禁 → 体验收口 → 技术债收口**
- 只有在代码实现且验证通过后，才能将任务状态标记为完成

### 0.2 优先级定义

- **P0**：直接影响系统稳定性、接口可信度、核心流程可验证性
- **P1**：影响质量持续保障、测试门禁与主流程体验
- **P2**：影响长期维护效率、文档沉淀与边缘优化

## 1. Phase 1 - 稳定性基线收口（P0）

### 1.1 版本与依赖治理

- [ ] **梳理 sklearn 版本不一致风险清单**
  - 输出所有涉及模型反序列化与加载的文件列表
  - 标记训练版本与推理版本可能不一致的位置
  - 产出依赖风险清单

- [ ] **统一 sklearn 训练/推理版本策略**
  - 固定版本范围或明确兼容范围
  - 必要时重新导出模型
  - 验证加载 warning 是否消失或被明确接管

- [ ] **替换所有 `datetime.utcnow()` 调用**
  - 使用 timezone-aware UTC 时间写法
  - 确保业务逻辑和测试均通过

- [ ] **治理全空特征 warning**
  - 定义全空列处理策略
  - 在预处理阶段统一处理或拦截
  - 验证 warning 数量下降

- [ ] **整理 PyTorch / Transformers 可选依赖行为**
  - 明确缺依赖时的 fallback 规则
  - 验证 Windows / Linux 环境行为一致性

### 1.2 边界与回退治理

- [ ] **统一输入边界处理口径**
  - 处理空值、全空列、缺字段、非法类型、极端分布、高延迟输入
  - 明确每类输入的响应策略

- [ ] **统一错误语义与返回结构**
  - 定义模型层、服务层、前端层一致的错误格式
  - 避免不同模块返回不同风格的错误

- [ ] **收口 fallback 分级机制**
  - 主模型失败
  - 融合模型失败
  - 规则回退
  - 启发式兜底
  - 每层输出统一日志

- [ ] **补充退化场景验证用例**
  - 模型不可用
  - 特征缺失
  - 低质量输入
  - 监控模块异常
  - 报告生成异常

### 1.3 稳定性验证

- [ ] **为稳定性治理新增回归测试**
  - 覆盖版本治理、边界治理、fallback 行为
  - 每个问题修复后必须有对应测试

- [ ] **执行稳定性全量回归**
  - 后端全量测试
  - 前端相关测试

## 8. 任务依赖顺序

### 8.1 强制顺序

1. 稳定性治理
2. 契约测试基础设施
3. E2E 测试基础设施
4. 契约 / E2E 核心用例实现
5. 覆盖率与测试基线固化
6. 质量工具链接入
7. 前端体验闭环优化
8. 技术债收口与文档沉淀

### 8.2 不建议跳跃的原因

- 没有稳定性治理，契约测试结果会受到版本/依赖干扰
- 没有契约测试基础设施，E2E 很容易只覆盖表面流程
- 没有覆盖率基线，后续改动无法判断是否回归
- 没有质量门禁，CI/CD 无法真正发挥阻断作用
