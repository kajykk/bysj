## 1. 迭代背景

v1.5 迭代已完成深度学习工程化安全底座（验证闭环、灰度发布、监控告警）和前端性能与数据可视化增强。v1.6 迭代聚焦于**稳定性治理与质量闭环收口**，在 v1.5 的基础上：

2. **建立契约测试** - 确保 API 和模型接口的稳定性与可信度
3. **建立 E2E 测试** - 确保核心用户流程端到端可验证
4. **固化测试基线** - 提升覆盖率，建立质量门禁
5. **质量工具链** - Lighthouse CI、Sentry、Codecov 接入
6. **前端体验闭环** - 大列表加载、报告导出稳定性、图表规范
7. **技术债收口** - Warning 清理、类型安全、经验沉淀

## 3. 详细需求

### 3.1 Phase 1 - 稳定性基线收口 (P0)

#### 3.1.1 版本与依赖治理
- **梳理 sklearn 版本不一致风险清单**
  - 输出所有涉及模型反序列化与加载的文件列表
  - 标记训练版本与推理版本可能不一致的位置
  - 产出依赖风险清单

- **统一 sklearn 训练/推理版本策略**
  - 固定版本范围或明确兼容范围
  - 必要时重新导出模型
  - 验证加载 warning 是否消失或被明确接管

- **替换所有 `datetime.utcnow()` 调用**
  - 使用 timezone-aware UTC 时间写法
  - 确保业务逻辑和测试均通过

- **治理全空特征 warning**
  - 定义全空列处理策略
  - 在预处理阶段统一处理或拦截
  - 验证 warning 数量下降

- **整理 PyTorch / Transformers 可选依赖行为**
  - 明确缺依赖时的 fallback 规则
  - 验证 Windows / Linux 环境行为一致性

#### 3.1.2 边界与回退治理
- **统一输入边界处理口径**
  - 处理空值、全空列、缺字段、非法类型、极端分布、高延迟输入
  - 明确每类输入的响应策略

- **统一错误语义与返回结构**
  - 定义模型层、服务层、前端层一致的错误格式
  - 避免不同模块返回不同风格的错误

- **收口 fallback 分级机制**
  - 主模型失败 → 融合模型失败 → 规则回退 → 启发式兜底
  - 每层输出统一日志

- **补充退化场景验证用例**
  - 模型不可用、特征缺失、低质量输入、监控模块异常、报告生成异常

#### 3.1.3 稳定性验证
- **为稳定性治理新增回归测试**
  - 覆盖版本治理、边界治理、fallback 行为
  - 每个问题修复后必须有对应测试

- **执行稳定性全量回归**
  - 后端全量测试、前端相关测试、检查 warning 变化

### 3.2 Phase 2 - 契约测试基础设施 (P0)

#### 3.2.1 环境与依赖准备
- 安装 schemathesis 和 hypothesis
- 创建 `backend/tests/contract/` 目录结构
- 导出并固化 OpenAPI Spec

#### 3.2.2 API 契约测试
- 认证 API、用户 API、模型预测 API、监控 API、灰度发布 API、验证与报告 API、报告导出 API
- 校验请求/响应结构、状态码、字段稳定性、异常输入时的结构化错误返回

#### 3.2.3 模型接口契约测试
- 表格预测、文本预测、生理数据预测、融合预测
- 校验输入字段完整性、输出格式、缺失字段回退、边界行为

### 3.3 Phase 3 - E2E 测试基础设施与核心流程 (P0)

#### 3.3.1 E2E 基础设施
- 安装并配置 Playwright
- 建立 E2E 测试目录结构
- 实现 Page Object 基类

#### 3.3.2 核心 Page Objects
- LoginPage、HomePage、RiskAssessmentPage、MonitoringDashboardPage、ReportCenterPage

#### 3.3.3 核心业务流程 E2E
- 用户登录流程、风险评估流程、监控面板流程、灰度发布流程、报告导出流程

### 3.4 Phase 4 - 覆盖率提升与测试基线固化 (P1)

#### 3.4.1 后端覆盖率提升
- 提升 `app/services/*`、`app/api/v1/*`、`app/core/*`、`app/ml/*` 覆盖率到目标

#### 3.4.2 前端覆盖率提升
- 提升 `src/components/*`、`src/composables/*`、`src/views/*` 覆盖率到目标

#### 3.4.3 测试基线固化
- 区分测试分层（单元/集成/回归/性能）
- 建立关键路径测试清单
- 建立质量门禁规则（PR 合并条件、覆盖率不下降、契约/E2E 必须通过）

### 3.5 Phase 5 - 质量工具链与持续治理 (P1)

#### 3.5.1 CI/CD 流水线
- 配置契约测试、E2E 测试、覆盖率、Lighthouse 流水线

#### 3.5.2 Sentry 与可观测集成
- 接入后端 Sentry 异常追踪
- 接入前端 Sentry 错误捕获和性能追踪

#### 3.5.3 覆盖率平台
- 接入 Codecov / Coveralls
- 配置 PR 覆盖率检查

#### 3.5.4 质量门禁
- 建立 PR 质量检查规则（契约/E2E/覆盖率/Lighthouse）

### 3.6 Phase 6 - 前端体验与报告闭环优化 (P1)

- **优化大列表和复杂页面加载体验** - 减少首屏等待、优化骨架屏
- **收口 PDF 报告导出稳定性** - 校验字段一致性、异常输入返回结构
- **收口 Excel 导出稳定性** - 校验批量导出格式、空数据与缺字段场景
- **统一图表与 Dashboard 展示规范** - 风险趋势图、模型性能面板、系统健康监控、数字格式统一
- **复查移动端基础体验** - 响应式布局、关键页面适配、小屏可用性

### 3.7 Phase 7 - 技术债收口与经验沉淀 (P2)

#### 3.7.1 Warning 收口
- 清理 sklearn 版本 warning、utcnow deprecation warning、全空特征 warning、PyTorch 兼容性 warning

#### 3.7.2 类型安全提升
- 减少测试文件中的 any、收紧 API 响应类型定义、收紧组件 props 类型

#### 3.7.3 经验沉淀
- 整理惰性导入策略经验、FastAPI 参数顺序修复经验、契约测试前置经验、E2E 容器化经验、覆盖率持续跟踪经验

#### 3.7.4 文档收口
- 完善依赖与环境说明文档、输出技术债清单、输出经验教训文档

## 4. 非功能需求

### 4.1 性能要求
- E2E 测试执行时间 < 10 分钟
- 契约测试执行时间 < 5 分钟
- 覆盖率报告生成时间 < 2 分钟

### 4.2 可维护性
- 测试代码遵循 Page Object Model (POM) 模式
- 契约测试用例可自动更新
- 覆盖率报告可在线查看

### 4.3 兼容性
- E2E 测试支持 Windows / macOS / Linux
- 契约测试支持 OpenAPI 3.0+

## 5. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| E2E 测试环境不稳定 | 高 | 使用 Docker 容器化测试环境 |
| 契约测试维护成本高 | 中 | 自动生成契约测试用例 |
| 覆盖率提升困难 | 中 | 优先覆盖核心业务流程 |
| 版本治理影响现有模型 | 高 | 逐步迁移，保留兼容性 |

## 7. 任务依赖顺序

### 强制顺序
1. 稳定性治理（Phase 1）
2. 契约测试基础设施（Phase 2）
3. E2E 测试基础设施（Phase 3）
4. 契约 / E2E 核心用例实现
5. 覆盖率与测试基线固化（Phase 4）
6. 质量工具链接入（Phase 5）
7. 前端体验闭环优化（Phase 6）
8. 技术债收口与文档沉淀（Phase 7）

### 不建议跳跃的原因
- 没有稳定性治理，契约测试结果会受到版本/依赖干扰
- 没有契约测试基础设施，E2E 很容易只覆盖表面流程
- 没有覆盖率基线，后续改动无法判断是否回归
- 没有质量门禁，CI/CD 无法真正发挥阻断作用
